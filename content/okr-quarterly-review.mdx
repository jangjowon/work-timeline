이번 분기에 설정한 OKR을 돌아보며 실제 달성도와 다음 분기에 이어갈 액션 아이템을 정리했다. 측정 지표가 모호했던 항목은 구체적인 메트릭으로 다시 정의해 다음 사이클에서 더 정확히 추적할 수 있도록 했다.

## 이번 분기의 Objectives

분기 시작 시점에 팀과 합의한 세 가지 큰 목표를 먼저 다시 적어본다. 각 Objective는 **성공했는지 여부를 한 문장으로 말할 수 있어야 한다**는 기준을 세웠다.

1. 사용자 유입 경로 전반의 관측성을 확보한다
2. 온보딩 이탈률을 절반으로 줄인다
3. 팀 내부 문서 체계를 실제로 굴러가는 자산으로 만든다

두 번째 항목처럼 숫자가 붙은 것은 리뷰가 쉬웠지만, 첫 번째와 세 번째는 "얼마나 했다"를 말하기가 생각보다 까다로웠다. 이 지점이 다음 분기 개선 포인트가 됐다.

### Key Results 달성도

실제 수치를 표로 정리하기 전에, 지표가 움직인 모습을 한눈에 보려고 대시보드 하나를 붙여두었다.

![분기 OKR 대시보드 스냅샷](/content-images/okr-dashboard.svg)

숫자만 보면 괜찮아 보이는 지표도 맥락을 붙여 설명하지 않으면 다음 분기에 동일한 판단 실수를 반복하기 쉽다. 그래서 각 KR마다 "왜 이 숫자가 나왔는지" 한 줄을 붙였다.

- **KR1.** 핵심 퍼널 5단계 중 4단계에 이벤트 추적 적용 — 나머지 1단계는 외부 리다이렉트라 다음 분기 이월
- **KR2.** 신규 사용자 7일 리텐션 28% → 41% — 온보딩 체크리스트 리뉴얼이 주효했음
- **KR3.** 내부 문서 재방문률이 측정 기준 미비로 판정 불가 — 다음 분기에 지표 정의부터 다시

## 측정 지표 재정의

달성도를 말하기 애매했던 항목들은 `measurable`하지 않았다는 점이 공통이었다. 아래처럼 수치와 판정 기준을 같이 적어두는 포맷으로 KR을 다시 썼다.

```ts
type KeyResult = {
  description: string;
  metric: string;        // 측정 도구 · 쿼리
  target: number;        // 수치 목표
  evaluatedAt: string;   // 판정 시점
};
```

특히 `evaluatedAt`을 강제하기로 한 게 큰 변화였다. "분기말 어느 시점에 어떤 쿼리로 봤다"를 명시해두지 않으면, 같은 KR을 놓고도 리뷰마다 다른 숫자가 나와 논쟁이 길어졌다.

> 측정할 수 없는 목표는 목표가 아니라 바람이다. 숫자와 판정 시점을 같이 적지 않은 KR은 다음 분기에 쓰지 않기로 했다.

## 다음 분기에 이어갈 액션

이번 리뷰에서 "이건 지금 바로 옮겨놓자" 싶었던 항목만 추려 다음 분기 초안에 붙여두었다.

1. 외부 리다이렉트 구간 추적 방안 설계 (`KR1` 이월분)
2. 온보딩 개편 효과의 재측정 — 리뉴얼 효과인지 계절성인지 분리
3. 문서 재방문률 정의와 대시보드 신규 구축

리뷰가 끝날 즈음에 느낀 건, OKR 회고가 결국 **다음 분기 목표 문장을 얼마나 좋게 다듬느냐**의 싸움이라는 것이었다. 이번에 정리한 포맷을 다음 사이클에서 한 번 더 돌려보고, 그때도 쓸 만하면 팀 템플릿으로 굳힐 생각이다.
