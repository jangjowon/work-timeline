# 포스트 작성 체크리스트

새 글을 발행하기 전에 한 번씩 훑어보는 체크리스트. `npm run new-post` 으로 만든 빈 mdx 파일을
이 체크리스트 위에서 채워나가면 된다.

---

## 1. 작성 전 확인

- [ ] **환자 / 내담자 / 타인 식별 가능한 정보가 모두 익명화되어 있는가**
  - 이름, 나이, 거주지, 직장, 진료 일자처럼 조합하면 특정인이 떠오르는 정보는 변형하거나 제거.
  - 사례를 합칠 수 있다면 합치고, 본질이 흐려지지 않는 선에서 디테일을 바꾸는 것이 안전하다.
- [ ] **소속 기관의 비공개 자료를 사용하지 않는가**
  - 보건소·병원·연구실의 내부 슬라이드, 환자 사진, 미공개 임상 데이터, 차트 캡처 등은 공개 가능 여부를
    먼저 확인하기 전에는 본문에 넣지 않는다.
  - 그래프나 표가 필요하면 직접 모형 데이터로 다시 그리거나, 출처를 밝힐 수 있는 공개 자료로 교체한다.
- [ ] **회사 / 기관 NDA·내규 범위 안에 있는가**
  - 진행 중인 연구의 미발표 결과, 계약/협약 내용, 인사 관련 정보는 발행 전에 한 번 더 본다.
- [ ] **외부 인용에 출처를 달았는가**
  - 논문·기사·공식 가이드라인은 링크 또는 서지정보로. 이미지 사용 시 라이선스를 확인.
- [ ] **AI / 보조 도구로 생성한 부분이 있다면 본인의 판단으로 검수했는가**

## 2. Frontmatter 필수 필드

```yaml
---
title: 분기 OKR 리뷰 정리          # 필수. 화면·OG·RSS·sitemap에 그대로 사용
date: 2026-04-25                  # 필수. YYYY-MM-DD (KST 기준). new-post가 자동 입력
tags:                             # 선택. 비워두려면 [] 로
  - 회고
  - OKR
cover: /content-images/foo.jpg    # 선택. 비워두려면 '' 로
---
```

- `title`, `date` 가 없으면 빌드 시 매니페스트에서 누락되어 글이 보이지 않는다.
- `date` 는 미래 날짜도 허용되지만, 정렬은 내림차순(`date` 큰 순)이라는 점만 기억하면 된다.
- `cover` 는 `/content-images/...` 로 시작하는 절대 경로를 권장.

---

## 3. 본문 작성 가이드

### 길이

- 권장 길이: **2~5문단**.
- 너무 길어지면 절을 나누거나 후속 글로 분리. 메인 페이지 발췌가 깔끔해진다.

### 이미지

- 본문 이미지는 `public/content-images/` 아래에 두고 `![alt](/content-images/foo.jpg)` 형식으로 참조.
- **alt 작성 규칙**
  - 이미지가 본문에 정보를 추가한다면, 그 정보를 짧게 풀어 쓴다.
    예) `![분기 OKR 대시보드 스냅샷](...)` → 무엇을 보여주는 캡처인지 알 수 있게.
  - 단순 장식이면 빈 alt(`![](...)`) 도 가능하지만, 가급적 의미 있는 설명을 권장.
  - 환자/검사 결과 등 민감한 이미지는 alt에도 식별 정보가 들어가지 않도록 주의.

### 코드, 인용, 수식

- 코드 블록은 언어 태그 포함 (` ```ts `, ` ```bash `).
- 긴 인용은 `>` 로 한두 줄. 본인의 생각과 인용을 구분.

---

## 4. 태그 작명 컨벤션

- 한국어 명사 위주 (예: `회고`, `진료`, `연구노트`, `독서`, `도구`).
- 영어 약어는 그대로 (`OKR`, `EMR`, `NLP`).
- 한 글에 **2~4개** 가 적당. 모두에게 붙는 광범위한 태그(`일상`)는 가급적 피한다.
- 새 태그를 만들 때는 기존 태그 목록과 너무 가깝지 않은지 한 번 본다.
